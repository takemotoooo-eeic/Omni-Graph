# RAGAnything Knowledge Graph App 仕様書

## 1. 目的

`raganything` を用いてドキュメントを取り込み、知識グラフを構築・可視化し、同一グラフに対して質問応答できる Streamlit アプリを `app` 配下に実装する。

本仕様は MVP（最小実用版）を対象とし、以下を実現する。

- ドキュメント入力と知識グラフ構築
- 構築済みグラフの可視化
- グラフを利用したチャット応答

## 2. 対象範囲

### 2.1 スコープ内

- Streamlit ベースの単一アプリ
- 3 タブ UI
  - ドキュメント入力
  - グラフ可視化
  - チャット
- `raganything` を利用した処理連携
- ローカル実行を前提とした保存領域（永続化）

### 2.2 スコープ外（将来拡張）

- ユーザー認証/マルチテナント
- 本番運用向け監視基盤
- 複数ユーザー同時編集の整合性管理

## 3. 前提・依存

- Python 3.10+（既存プロジェクト準拠）
- `streamlit`
- `raganything`
- グラフ描画ライブラリ（候補: `pyvis` または `plotly` + `networkx`）
- LLM/API 接続に必要な環境変数（例: API キー）

## 4. 全体アーキテクチャ

### 4.1 論理構成

- UI 層: Streamlit（3 タブ）
- アプリケーション層:
  - ドキュメント処理サービス
  - グラフサービス
  - チャットサービス
- データ層:
  - 入力ドキュメント
  - 抽出済みノード/エッジ
  - セッション別チャット履歴

### 4.2 処理フロー

1. ユーザーがドキュメントをアップロード
2. `raganything` でエンティティ/関係を抽出
3. ノード・エッジを保存し、知識グラフを更新
4. 可視化タブでグラフを描画
5. チャットタブで質問を受け取り、グラフ情報を根拠に回答生成

## 5. 画面仕様（Streamlit タブ）

## 5.1 タブ1: ドキュメント入力

### 目的

知識グラフ構築のための入力データを登録・処理する。

### UI 要素

- ファイルアップローダー（複数可）
- 対応拡張子表示（例: `pdf`, `txt`, `md`, `docx`）
- 「グラフ構築開始」ボタン
- 処理進捗表示（スピナー/プログレス）
- 結果サマリ表示
  - 処理成功件数/失敗件数
  - 生成ノード数/エッジ数
  - エラー詳細（失敗時）

### 振る舞い

- ファイル未選択時は構築ボタンを無効化
- 構築中は二重実行を防止
- 処理完了後にグラフ可視化タブへ遷移可能な案内を表示

## 5.2 タブ2: グラフ可視化

### 目的

構築した知識グラフの全体構造を理解できる状態で提示する。

### UI 要素

- グラフ表示領域（インタラクティブ）
- 表示フィルター
  - ノードタイプ
  - 関係タイプ
  - 最小エッジ重み
- レイアウト切替（例: force/circular）
- ノード詳細パネル
  - ID
  - ラベル
  - 属性（出典ドキュメント等）

### 振る舞い

- グラフ未生成時はプレースホルダーメッセージを表示
- ノードクリックで隣接ノード/関係をハイライト
- フィルター変更時に再描画

## 5.3 タブ3: チャット

### 目的

知識グラフを参照した質問応答を行う。

### UI 要素

- チャット履歴表示
- ユーザー入力ボックス
- 送信ボタン
- 応答補助情報（任意）
  - 参照ノード数
  - 主要参照エッジ
  - 根拠ドキュメント

### 振る舞い

- グラフ未生成時は質問不可のガイド表示
- 送信中は入力を一時無効化
- 回答失敗時は再試行案内とエラーメッセージ表示

## 6. 機能要件

### FR-01 ドキュメント取り込み

- ユーザーは 1 件以上のドキュメントをアップロードできる
- 対応外形式は入力時点で拒否する

### FR-02 知識グラフ構築

- 入力ドキュメントから `raganything` 経由で知識を抽出する
- ノード・エッジをアプリ内部形式に変換し保持する

### FR-03 グラフ可視化

- ノード/エッジの関係を視覚的に表示する
- フィルター/ハイライトに対応する

### FR-04 グラフ QA チャット

- ユーザー質問に対し、グラフ情報を用いて回答を生成する
- 回答時に可能な範囲で根拠情報を提示する

### FR-05 セッション管理

- Streamlit セッション単位で処理状態を保持する
- チャット履歴を同セッション内で保持する

## 7. 非機能要件

### NFR-01 性能

- 10MB 程度の単一 PDF を初回処理する際、待ち時間が長い場合でも進捗が見えること
- 可視化はユーザー操作に対し体感上遅延が少ないこと

### NFR-02 可用性・耐障害性

- API エラー時にアプリが停止せず、UI で原因を通知すること
- 不正入力に対しクラッシュしないこと

### NFR-03 保守性

- UI ロジックと `raganything` 呼び出しロジックを分離する
- 主要処理は関数/モジュール単位でテスト可能にする

### NFR-04 セキュリティ

- API キーは環境変数で管理し、ログへ出力しない
- アップロードファイルは想定拡張子とサイズ上限を検証する

## 8. データ仕様

### 8.1 ドキュメントメタデータ

- `document_id`
- `filename`
- `uploaded_at`
- `status`

### 8.2 ノード

- `node_id`
- `label`
- `type`
- `source_document_ids`
- `attributes`（任意）

### 8.3 エッジ

- `edge_id`
- `source_node_id`
- `target_node_id`
- `relation`
- `weight`（任意）
- `source_document_ids`

### 8.4 チャットメッセージ

- `message_id`
- `role`（user/assistant）
- `content`
- `timestamp`
- `references`（任意）

## 9. エラーハンドリング方針

- 入力エラー: UI 上で具体的に修正方法を提示
- 処理エラー: リトライ可能な失敗と不可逆失敗を区別して表示
- 外部 API エラー: ステータスと要約メッセージを表示

## 10. ログ・観測性

- 最低限のイベントログ
  - アップロード開始/完了
  - グラフ構築開始/完了/失敗
  - チャット質問/応答失敗
- 個人情報や API キーはログ出力しない

## 11. ディレクトリ案（`app` 配下）

- `app/main.py`（Streamlit エントリ）
- `app/ui/`（タブ UI）
- `app/services/`（ドキュメント処理、グラフ、チャット）
- `app/models/`（データモデル）
- `app/utils/`（共通処理）

## 12. 受け入れ基準（MVP）

- [ ] 3 タブ構成で画面が表示される
- [ ] ドキュメント入力から知識グラフが 1 回以上正常に構築できる
- [ ] 可視化タブでノード/エッジが表示される
- [ ] チャットで質問し、回答が表示される
- [ ] 主要エラー時に UI でユーザーが原因を把握できる

## 13. 将来拡張案

- グラフのエクスポート（JSON/GraphML）
- 回答根拠の可視リンク（ノード選択と連動）
- ドキュメント差分更新（再取り込み最適化）
- ユーザーごとのワークスペース分離
